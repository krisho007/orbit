# Architecture

## How It All Fits Together

```
┌─────────────────────┐      ┌────────────────────────────────────┐
│  Browser / PWA      │─────▶│  Fly.io Container                  │
│  (installable)      │      │  ┌──────────────────────────────┐  │
└─────────────────────┘      │  │  Hono API   (/api/*)         │  │
                             │  ├──────────────────────────────┤  │
                             │  │  Expo Web   (/* static, PWA) │  │
                             │  └──────────────────────────────┘  │
                             └──────────────┬─────────────────────┘
                                            │
                                            ▼
                             ┌──────────────────────────────┐
                             │  Neon (PostgreSQL)            │
                             └──────────────────────────────┘
```

- The **API** (Hono on Bun) serves REST endpoints at `/api/*`
- The **Web app** (Expo Web static export) is bundled into the same Docker image and served at `/` by Hono's `serveStatic`, as an installable **PWA** (service worker + manifest). It is the only client — there are no native (Android/iOS) builds.
- Auth is handled by **Better Auth** (Google OAuth); it owns `/api/auth/*`, and the database is **Neon** Postgres

## API Layer (`apps/api`)

### Entry Point

`apps/api/src/index.ts` - Creates Hono app, registers middleware (CORS, logger), mounts route modules, serves static Expo Web files as SPA fallback.

### Route Pattern

Each domain has its own route file in `apps/api/src/routes/`:
- `contacts.ts`, `conversations.ts`, `events.ts`, `reminders.ts`, `tags.ts`, `relationships.ts`, `users.ts`, `speech.ts`
- `assistant/` - Complex sub-module with its own directory structure

Better Auth owns `/api/auth/*` (sign-in, Google OAuth callback, session, sign-out), mounted in `index.ts` via `auth.handler`. Other routes use `authMiddleware` from `apps/api/src/middleware/auth.ts` to protect endpoints. The middleware:
1. Calls `auth.api.getSession({ headers })`, which accepts either the web session cookie or a native `Authorization: Bearer` token
2. Rejects with 401 if there is no valid session
3. Sets `c.set("user", ...)` and `c.set("userId", session.user.id)` on the Hono context

### Data Layer

- **DB client**: `apps/api/src/db/index.ts` - Drizzle ORM instance with postgres.js driver
- **Schema**: `apps/api/src/db/schema.ts` - All table definitions, enums, relations, and type exports
- **Migrations**: `apps/api/drizzle/` - SQL migration files generated by `drizzle-kit`
- Connection pooling via Neon's pooled connection endpoint

### Database Models

Core models (all have `userId` for multi-tenancy):

| Model | Description | Key Relations |
|-------|-------------|---------------|
| `contacts` | People in network | tags, images, socialLinks, conversations, events, reminders via participant tables |
| `conversations` | Interaction records | participants (contacts), linked event, reminders |
| `events` | Calendar events | participants (contacts), linked conversations |
| `reminders` | Follow-up tasks | participants (contacts), linked conversation |
| `tags` | Labels for contacts | many-to-many via `contact_tags` |
| `relationships` | Contact-to-contact links | typed via `relationship_types` (directional, supports gender-aware reverse types) |
| `contactImages` | Photos | linked to contact |
| `socialLinks` | Social media URLs | linked to contact |

Participant tables (`conversation_participants`, `event_participants`, `reminder_participants`) create many-to-many links between entities and contacts.

### AI Assistant (`apps/api/src/routes/assistant/`)

The assistant uses Google Gemini via Vercel AI SDK (`@ai-sdk/google`) with function calling:

- `route.ts` / `index.ts` - Hono route handler
- `process-message.ts` - Core message processing with Gemini
- `system-prompt.ts` - Intent-based system prompt generation
- `tools/` - Function calling tools (contacts, conversations, events, reminders, tags, relationships)
- `ui-builder.ts` - Builds structured UI cards for the mobile app to render
- `guardrails.ts` - Input classification and safety
- `ownership.ts` - Verifies userId ownership before mutations
- `enrichment.ts` - Enriches tool results with related data
- `db-helpers.ts` - Database query helpers for the assistant

## Web App Layer (`apps/app`)

Built with Expo (React Native Web) + Expo Router and exported as a static web bundle (`output: "static"`). See the [web app guide](web.md) for the full route map and UI conventions.

### PWA

The web export is shipped as an installable PWA:
- **Manifest + icons**: `apps/app/public/manifest.json` and `apps/app/public/icons/*` (copied to the export root).
- **Root HTML**: `apps/app/app/+html.tsx` injects the manifest/meta tags and registers the service worker.
- **Service worker**: generated at build time by Workbox (`apps/app/workbox-config.js`, run after `expo export`). It precaches the content-hashed bundles (fast, offline app shell via `navigateFallback`), network-firsts `/api` GETs, and uses `skipWaiting`/`clientsClaim` so a new deploy is picked up on the next load.
- **HTTP caching** (set in `apps/api/src/index.ts`): content-hashed assets under `/_expo/static/*` are `immutable`; HTML, `sw.js`, and `manifest.json` are `no-cache` so a UI change busts the cache.

### Auth Flow

`apps/app/lib/auth.tsx` provides `AuthProvider` and `useAuth()`: standard OAuth redirect with PKCE code exchange, then a session established with the API.

### API Client

`apps/app/lib/api.ts` provides:
- `ApiClient` class that auto-attaches the auth token to every request
- Domain-specific API methods: `contactsApi`, `conversationsApi`, `eventsApi`, `remindersApi`, `tagsApi`, `assistantApi`, `speechApi`
- All TypeScript types for API payloads and responses

### Theme System

`apps/app/lib/theme.tsx` provides:
- `ThemeModeProvider` / `useThemeMode()` - Light/dark mode with AsyncStorage persistence
- `useThemeColors()` / `getThemeColor()` - Color token resolution
- Gluestack UI provider wraps the app for component theming

### Onboarding

`apps/app/lib/onboarding.ts` manages versioned onboarding state via AsyncStorage. The root layout checks onboarding status and redirects new users to the welcome flow.

## Environment Variables

### API (`apps/api/.env`)

| Variable | Required | Description |
|----------|----------|-------------|
| `DATABASE_URL` | Yes | Neon Postgres connection string (pooled, `?sslmode=require`) |
| `BETTER_AUTH_SECRET` | Yes | Session signing secret (`openssl rand -base64 32`) |
| `BETTER_AUTH_URL` | Yes | Public API base URL (OAuth callback + image URLs) |
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` | Yes | Google OAuth (sign-in + Contacts import) |
| `GOOGLE_GENERATIVE_AI_API_KEY` | For assistant | Google AI Studio API key for Gemini |
| `CORS_ALLOWED_ORIGINS` | No | Comma-separated allowed origins |
| `PORT` | No | Server port (default: 3001) |

### Web app (`apps/app/.env`)

| Variable | Required | Description |
|----------|----------|-------------|
| `EXPO_PUBLIC_SUPABASE_URL` | Yes | Supabase project URL |
| `EXPO_PUBLIC_SUPABASE_ANON_KEY` | Yes | Supabase anonymous key |
| `EXPO_PUBLIC_API_URL` | Dev only | API URL (empty on prod web = same origin) |

Note: Expo requires the `EXPO_PUBLIC_` prefix for client-accessible env vars. Bun auto-loads `.env` files.
